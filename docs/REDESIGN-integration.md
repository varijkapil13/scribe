## Notes editor & reading experience

**Deferred (needs spine):**
- Per-note typeface persistence via NoteFrontmatter.setExtra("font", ...): the Phase-0 frontmatter unknown-key passthrough (.extra / setExtra / extraValue) does NOT exist in this worktree — NoteFrontmatter/NoteFrontmatterCodec are the pre-Phase-0 shape and are storage-layer files outside my allow-list. To avoid silent data loss and out-of-scope edits I persisted the per-note font app-side (@AppStorage keyed by note id). This does NOT survive an external Obsidian edit / vault move and is not shared cross-device. Migrating to frontmatter is a clean follow-up once the passthrough lands (see integration hooks).
- View-menu shortcut to toggle page width Regular/Wide: the in-editor overflow menu and shortcut are mine, but a global View-menu CommandGroup item belongs to the spine command tree (ScribeApp/menu tree), not an editable file (see integration hooks). The ^cmd+F focus-mode shortcut is wired locally on NoteDetailView's toolbar button; promoting it to the View menu is also a spine task.
- Spatial card push/pop + breadcrumb for wiki-link navigation: explicitly deferred per the brief — needs NavigationCoordinator in MainWindowView/NotesBrowserView (spine-owned). Wiki-link navigation still works via the existing onNavigate teleport.
- NOTE on assumed environment: the brief states a Phase-0 DesignTokens substrate exists (scribeGlass, Motion.snappy/.resolve, scribeAnimation, scribeAccent, contrast-aware Palette helpers, prioritySymbol*, NoteFrontmatter.extra). NONE of these are present at this HEAD (branch is master at the markdown-AST commit, not redesign/iina-craft-arc). I therefore used the accessibility-equivalent primitives that DO exist: @Environment(\.accessibilityReduceTransparency/ReduceMotion/colorSchemeContrast) with .regularMaterial -> DesignTokens.Palette.surfaceElevated solid fallbacks, and DesignTokens.Motion durations via withAnimation gated on Reduce Motion. When the Phase-0 substrate is merged, the material/animation/contrast call sites in SlashCommandMenu.swift, FormatBubble.swift, and NoteEditorView.swift should be swapped to .scribeGlass(.hud) / .scribeAnimation / accentFill for consistency.

**Integration hooks:**
- View menu (ScribeApp.swift CommandGroup / menu tree — spine-owned): add a 'Page Width' toggle that flips @AppStorage("noteEditor.pageWidth") between NotePageWidth.regular/.wide. Suggested: read AppStorage("noteEditor.pageWidth"), and on the menu action set it to NotePageWidth(rawValue: current)?.toggled.rawValue ?? NotePageWidth.regular.rawValue. A non-reserved chord like ^cmd+W collides with Close Window — prefer something like ^cmd+. or leave shortcutless. The keys/enum live in Scribe/UI/Notes/NoteEditorPreferences.swift (NotePageWidth.storageKey, .toggled).
- View menu: optionally promote Focus Mode to a real menu item bound to @AppStorage("noteEditor.focusMode") (Bool toggle). It is currently a NoteDetailView toolbar button with .keyboardShortcut("f", modifiers: [.control, .command]); a menu item would make it discoverable and let the spine own the shortcut-allocation table entry for ^cmd+F.
- Frontmatter migration for per-note typeface (once Phase-0 NoteFrontmatter.extra / setExtra(_: _:) / extraValue(forKey:) exists): in NoteDetailViewModel (or NoteStore save path), persist the chosen typeface as frontmatter setExtra("font", <NoteTypeface.rawValue>) and read it back via extraValue(forKey: "font"), then pass it down so NoteEditorView uses it instead of the @AppStorage-by-id store. The enum + raw values are in NoteEditorPreferences.swift (NoteTypeface system/serif/mono). This is what makes the choice survive external Obsidian edits and vault moves; until then app-side persistence is the safe fallback.
- If/when the Phase-0 design substrate (scribeGlass/.hud, scribeAnimation, accentFill, contrast-aware Palette) is merged: update the material + animation + contrast call sites in SlashCommandMenu.swift (menuBackground/borderColor/highlightFill), FormatBubble.swift (bubbleBackground/borderColor/transition), and NoteEditorView.swift (withAnimationIfAllowed, focusDimAlpha) to use them so the editor chrome matches the rest of the redesign. No API of mine needs to change — these are internal styling swaps.

## Live recording, transcription & floating controller

**Deferred (needs spine):**
- Placing LiveControllerOverlay as a MainWindowView .overlay (spine-owned MainWindowView.swift) — built standalone, hook described below.
- Presenting OnboardingView as a .sheet on first run (spine-owned MainWindowView.swift / ScribeApp.swift) — built standalone, hook described below.
- The Record->Live matchedGeometry morph — out of scope per task (needs the toolbar record control which lives in MainWindowView/spine).
- Wiring AudioBufferManager.onSystemLevel into AppState's wireAudioPipeline as the system-level source (AppState.swift is not in my allow-list). Not needed for functionality: AudioSessionManager already publishes systemLevel directly from systemCapture buffers, so the meter works without touching AppState. The AudioBufferManager.onSystemLevel callback is provided as an additive, ready-to-use hook for whoever owns AppState.

**Integration hooks:**
- In Scribe/UI/MainWindow/MainWindowView.swift: attach the floating controller as an overlay on the main content, e.g. add `.overlay { LiveControllerOverlay(audioManager: appState.audioManager, appState: appState, onPauseToggle: { if appState.audioManager.isPaused { Task { await appDelegate.resumeRecording() } } else { appDelegate.pauseRecording() } }, onStop: { Task { await appDelegate.stopRecording() } }, onExpand: { /* navigate to the live session surface, e.g. selection = .live or coordinator.navigate(to live) */ }) }`. The overlay self-gates on appState.isTranscribing, so it is inert when not recording. Pass the existing @EnvironmentObject appState/appDelegate.
- Present onboarding on first run from a spine view (MainWindowView.swift or ScribeApp.swift): add `@AppStorage("hasCompletedOnboarding") private var hasCompletedOnboarding = false` and `@State private var showOnboarding = false`, then `.sheet(isPresented: $showOnboarding) { OnboardingView(isPresented: $showOnboarding, audioManager: appState.audioManager) }` with `.onAppear { if !hasCompletedOnboarding { showOnboarding = true } }`. OnboardingView sets hasCompletedOnboarding = true itself on finish/skip and flips the binding to dismiss.
- (Optional) To feed the system-audio meter through AppState's existing AudioBufferManager instead of (or in addition to) AudioSessionManager's direct path, set `audioBufferManager.onSystemLevel = { [weak self] peak in /* forward to a published level */ }` in AppState.wireAudioPipeline. Not required — AudioSessionManager.systemLevel already drives the UI today.
